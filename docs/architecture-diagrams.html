Create URL Flow

Browser gateway ko request bhejta hai. Gateway abuse check ke baad optional JWT decode karta hai. URL Service Snowflake ID generate karke Base62 short code banata hai, shard choose karta hai, row insert karta hai, aur anonymous case me claim token return karta hai.

Redirect Flow

Gateway /r/:shortCode redirect service ko proxy karta hai. Redirect service pehle Redis cache check karti hai; miss hone par URL Service se original URL fetch hota hai. Saath me click event BullMQ queue me jata hai aur browser ko 302 milta hai.

Analytics Flow

Analytics worker queue se raw event uthata hai, IP aur user-agent enrich karta hai, Redis counters increment karta hai, aur batch me click_events table me save karta hai. Dashboard endpoint combined Redis + Postgres data return karta hai.

Abuse Protection Flow

Gateway har public route se pehle abuse service ko call karta hai. Abuse service pehle Bloom filter backed blacklist check karta hai, phir Redis token bucket rate limiter chalata hai, aur repeated violations par IP ko auto-blacklist kar sakta hai.

Why 3 shard tables?

url-service/src/shard-router.ts short code ke first character ke ASCII value ko 3 se modulo karke shard choose karta hai. Isliye ER me 3 alag physical tables dikhana sahi hai.

Why click_events → logical_url_union?

click_events.short_code kisi ek shard se relate karta hai, lekin direct SQL foreign key possible nahi hai kyunki 3 shard tables hain. Isliye diagram me logical union node use kiya gaya hai.

Why blacklisted_ips is marked separate?

Aapke Neon schema me table hai, lekin current abuse-service runtime Redis-based blacklist aur Bloom filter use karti hai. Matlab DB table abhi operational source of truth nahi lag raha.
